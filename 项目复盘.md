## 1. 这个项目是什么（业务）

一个企业内部的**出差申请与审批**系统，两种角色：

- **员工**：提交出差申请（目的地、起止日期、事由、预估费用），查看自己申请的审批状态。
- **主管**：看到待审批的申请，逐条通过或驳回，可附审批意见。

核心业务闭环：**员工提交 → 申请变“待审批” → 主管审批 → 状态流转为“已通过/已驳回” → 员工看到结果**。

> 为什么选它做第一个项目：它是 B 端应用最典型的骨架——**数据建模 + 增删改查 + 角色权限 + 审批状态机**，学会这一套，大多数企业内部系统都是它的变体。

## 2. 技术栈与选型理由

| 层 | 选型 | 为什么选它（用 PM 的话说） |
|---|---|---|
| 后端 | **Python + FastAPI** | Python 最易读、AI 写得最稳；FastAPI 自动生成交互式接口文档（Swagger），能直观看到“接口”长什么样；市面招后端最通用 |
| 数据建模 | **SQLModel** | 一个 Python 类同时定义“数据库表”和“接口数据格式”，不用写两遍 |
| 数据库 | **SQLite** | 一个文件（travel.db）就是数据库，零配置，最适合理解“数据怎么存” |
| 前端 | **原生 HTML/JS** | 不引入 React 等框架，每一行都能看懂，专注理解“前端怎么调后端接口” |
| 版本管理 | **git + GitHub** | 工程协作基本功，代码可追溯、可分享 |
| 部署 | **Render（Blueprint）** | 免费、对新手友好，连上 GitHub 仓库即可自动部署，拿到公网链接 |

**选型背后的判断（面试可讲）**：能用更简单的方案达到目的，就不上复杂的。第一个项目的目标是“跑通全链路 + 看懂每一块”，所以刻意选了最易读、配置最少的组合，而不是堆时髦框架。

## 3. 系统架构

### 餐厅类比（建立直觉）

| 角色 | 在本项目里 | 大白话 |
|---|---|---|
| **前端** | `frontend/index.html` | 顾客看到的菜单和餐桌——人点点点的界面 |

### 一次请求的完整链路

```
用户在网页点“提交” → 前端 fetch('/travel-requests', POST, 表单数据)
   → 后端 main.py 的 create_travel_request 函数接住
   → 校验数据格式（TravelRequestCreate）
   → 存进数据库 travel.db（status 默认 pending）
   → 返回带 id 和时间的完整记录 → 前端刷新“我的申请”列表
```

> 关键认知：**前端只管界面和发请求，后端只管业务和数据**。两者通过 API 用 JSON 对话。这就是“前后端分离”。

## 4. 数据模型（数据库里的两张表）

### User（用户表）

| 字段 | 类型 | 说明 |
|---|---|---|
| id | int | 主键，唯一编号（自动生成） |
| username | str | 登录账号，唯一 |
| password | str | 密码（教学项目存明文，真实项目须加密——见第 13 节） |
| name | str | 姓名 |
| role | enum | employee（员工）/ manager（主管） |

### TravelRequest（出差申请表）

| 字段 | 类型 | 说明 |
|---|---|---|
| id | int | 主键 |
| applicant_id | int | 申请人，**外键**指向 User.id |
| destination | str | 目的地 |
| start_date / end_date | date | 出差起止 |
| reason | str | 事由 |
| estimated_cost | float | 预估费用 |
| status | enum | pending（待审批）/ approved（已通过）/ rejected（已驳回），默认 pending |
| approver_id | int? | 审批人，外键指向 User.id |
| approval_comment | str? | 审批意见 |
| created_at | datetime | 提交时间，自动记录 |

> **主键 vs 外键**：主键 = 这张表每条记录的唯一身份证（id）；外键 = 指向另一张表的某条记录（applicant_id 指向某个用户），把两张表关联起来。

## 5. 接口清单（API）

| 方法 | 路径 | 作用 | 谁用 |
|---|---|---|---|
| GET | `/` | 健康检查（确认后端活着） | 系统 |
| POST | `/login` | 登录，校验账号密码 | 全部 |
| POST | `/travel-requests` | 提交一条出差申请 | 员工 |
| GET | `/travel-requests` | 查申请，可按 `status` / `applicant_id` 过滤 | 全部 |
| GET | `/travel-requests/{id}` | 查某一条申请 | 全部 |
| POST | `/travel-requests/{id}/approve` | 主管通过 | 主管 |
| POST | `/travel-requests/{id}/reject` | 主管驳回 | 主管 |

附带两个自动页面：`/docs`（接口操作台 Swagger）、`/ui/`（网页界面）。

> **GET 还是 POST？** 经验法则：只是“查/读”用 GET，会“改/写”数据用 POST。这是 REST 风格的基本约定。

## 6. 三个核心设计点（面试重点讲这个）

### ① 审批状态机
申请的状态只在 `待审批 → 已通过 / 已驳回` 之间单向流转，不能回退、不能跳变。
> 面试讲法：“我把审批抽象成一个状态机，明确每个状态能流转到哪、不能流转到哪。”

### ② 角色权限校验（返回 403）
审批前检查操作人的 `role` 是否为 manager，员工越权审批直接拦截。
> 面试讲法：“我做了角色权限，员工和主管能干的事不同，越权操作直接拒绝。”

### ③ 防重复审批（返回 400）
已审批（非 pending）的申请再次审批会被拒绝，避免一条申请被重复处理导致数据错乱。
> 面试讲法：“我加了状态保护，保证一条申请只能被有效处理一次——这是后端健壮性设计。”

> 这三点正好回答面试题“你会怎么设计一个审批流”。而且你不只说得出，是**亲手实现过**。

## 8. 我学到的概念（大白话速查）

| 概念 | 一句话理解 |
|---|---|
| 前端 / 后端 / 数据库 | 界面 / 业务逻辑和接口 / 存数据 |
| API（接口） | 前后端之间约定好的“传话窗口”，一个地址=一个能调的功能 |
| REST / HTTP 方法 | GET 读、POST 写，是接口的通用约定 |
| 状态码 | 200 成功、400 请求有问题、401 没登录、403 没权限、404 找不到 |
| JSON | 前后端之间传数据的通用格式（像结构化的表单） |
| 主键 / 外键 | 唯一身份证 / 指向别的表的关联 |
| ORM（SQLModel） | 用 Python 类操作数据库，不用手写 SQL |
| 状态机 | 用“状态 + 允许的流转”来建模业务流程（如审批） |
| 同源 / CORS | 前端和接口在同一个地址就不用配跨域——所以我把前端挂在后端上 |
| 虚拟环境（venv） | 给项目单独装依赖，不污染系统，`.venv` 文件夹就是 |
| git / commit / .gitignore | 代码时光机 / 一个存档点 / 哪些文件不纳入管理 |
| 部署 / 环境变量 `$PORT` | 把代码搬到云服务器跑；端口等由平台动态指定 |

## 9. 踩过的坑（复盘价值最高的部分）

| 现象 | 原因 | 解决 |
|---|---|---|
| 自检报错 `no such table: user`（表没建出来） | `TestClient(app)` 直接实例化时，FastAPI 的启动钩子（建表+灌数据）不会触发 | 用 `with TestClient(app) as client:` 包起来，启动逻辑才会执行 |
| 数据库文件位置混乱 | 用相对路径会跟着“从哪启动”变 | 在 database.py 里用 `__file__` 算出绝对路径，固定放在 backend 目录 |
| 部署时担心 .venv/数据库被传上去 | 这些不该进仓库 | `.gitignore` 排除 `.venv/` 和 `*.db`，只提交源码 |

> 经验：**遇到报错先看最后一行**（真正的错误原因），别被前面一大堆调用栈吓到。

## 13. 后续可优化（第二阶段 / 加分项）

- **安全**：密码明文 → 加密存储（如 bcrypt）；登录改成 token/会话鉴权，而不是每次传 id。
- **AI 增强**：一句话“下周一到三去上海见客户”自动解析成结构化申请单；自动生成出差总结。
- **功能**：申请可撤回/修改、审批多级流转、出差费用统计、邮件/站内通知。
- **数据**：SQLite → PostgreSQL（云上数据不重置）。
- **前端**：换成组件化框架、加移动端适配。

> 这些“没做的部分”不是缺点，面试时主动讲“我当前优先保证核心闭环跑通，这些是规划中的下一步”，反而体现你的取舍判断。
